NearbyNote

Location-based reminder app built with Kotlin and modern Android tools, focused on stable alerts in real-world conditions.

NearbyNote is a location-based reminder app that lets people pin notes to real-world places and get notified when they enter the area. Users can create notes from a Mapbox-powered map or by searching for an address, choose a radius for the reminder, and then view their notes either on the map or in a list. The app also supports favorite places (like home or work) for quick reuse and voice input for creating notes hands-free.

  • Tech stack Kotlin, Jetpack Compose, MVVM, Hilt, Room, Coroutines/Flow, Mapbox Maps SDK (v11), Mapbox Geocoding API, Google Play Services Location (Geofencing), Android SpeechRecognizer, Ktor + kotlinx.serialization, GitHub Actions
  • Architecture
    • MVVM with Hilt-injected repositories: Each screen has a ViewModel that exposes UI state as Compose state/Flows and delegates data work to repositories instead of touching Room or Mapbox directly.
    • Testable interfaces: Interfaces like GeofenceRepository and MapboxRepository, plus fakes in the test package, let me unit-test geofence and Mapbox logic without real Android services or network calls.
    • Navigation using NavGraphs: A single NavHost with a bottom navigation bar routes between note list, map, read/write note, and saved-address screens, using typed arguments (e.g., noteId: Long?) instead of manual intent extras.
    • Room as a single source of truth: One Room database storing notes, geofences, and saved addresses; DAOs expose Flows that feed into ViewModels and also power background workers, so geofence state can be rebuilt after reboots or app updates.
  • Reliability
    • Room-backed geofence state: Geofences are stored in Room instead of only in memory, so the app can always recover the full set of active geofences from the database.
    • Background reconcile worker: A geofence reconcile worker, triggered by boot and package-replaced broadcast receivers, re-registers geofences after device reboots or app/Play Services updates without relying on a foreground service.
    • Notification sync: Notifications are driven from the same database state and can be cancelled via a dedicated receiver, which helps prevent “ghost” reminders when geofences are edited or removed.
    • Permission handling: Location permissions, including background access, are requested only when needed with clear in-app explanations and a linked privacy policy, so users understand why location is used and how their data is kept private.
  • What I focused on
    • Real-world behavior over happy paths: Making location-based reminders behave predictably in real-world conditions instead of just the happy path — handling flaky GPS, permission toggles, app restarts, and background limits.
    • Structured async & background work: Using coroutines, Flows, and WorkManager in a structured way so background work is cancelable and tied to the right lifecycle, while Compose reads state from a single Room-backed source of truth.
    • Maintainable & testable code: Writing code I can reason about later: clear naming, small functions, and interfaces/fakes around Mapbox and geofencing so most of the behavior can be unit-tested without a device.
  • Testing & CI/CD
    • Unit tests for core logic: Unit tests around repositories and ViewModels for notes and geofences so I can verify business logic without relying on a device or emulator.
    • Automated CI on each change: A GitHub Actions workflow runs the unit test suite on each push and pull request, helping catch regressions in core features before changes ship.
    • Release pipeline to Google Play: A separate release workflow builds a signed release and uploads it to Google Play when I push to the release branch, giving me hands-on experience with a lightweight mobile CI/CD pipeline similar to a real team.

HealthFusion

A health management app that tracking workout data, built with modern android development tech stacks.

HealthFusion is a workout-tracking app that helps people log their workouts and stay consistent with their fitness goals. Users can add workouts with details like sets, reps, weights, or duration, tag them by type, and review their history through a calendar view and progress charts. The app also supports daily and weekly workout goals, offline-friendly logging with automatic cloud sync when online, and per-user accounts so data stays backed up and private.

  • Tech stack Kotlin, Jetpack Compose, MVVM, Hilt, Room, Coroutines/Flow, Firebase Firestore, Firebase Authentication, Ktor + kotlinx.serialization
  • Architecture
    • MVVM with Hilt-injected ViewModels: The workout feature has its own ViewModel that exposes UI state as StateFlow for Compose screens. Hilt provides the Room DAOs, FirestoreRepository, and NetworkHelper, so UI code never talks directly to Firebase or the database.
    • Room as the single source of truth: An AppDatabase stores workouts, workout goals, and goal details. DAOs return Flow objects that feed into the ViewModels, and Compose observes those flows so the UI is always driven from the same local database state, online or offline.
    • Offline-first sync layer with Firestore: Local changes are written to Room first and marked with isSynced and lastModified fields. Sync functions in the ViewModels convert entities to DTOs and use a shared FirestoreRepository to push unsynced data and pull remote changes, resolving conflicts based on which side has the newer timestamp.
    • Composable navigation graphs for auth & main app: AuthNavGraph handles login/sign-up, and a main NavGraph routes between workout screens. Navigation uses typed arguments (for example, passing a workout goal type into the goal-setting screen) instead of ad-hoc string extras, which keeps routing safer and easier to maintain.
  • Reliability
    • Room-backed workout history: All workouts and workout goals are stored in a Room database, and Compose screens always render from that local state, so logging and browsing history still work even if Firestore or the network is down.
    • Offline logging with deferred sync: New workouts are inserted into Room first with isSynced = false and a lastModified timestamp. When connectivity is available, the ViewModel finds unsynced rows and pushes them to Firestore, then marks them as synced so users don’t lose workouts just because they were offline.
    • Network-aware cloud operations & safe deletes: A NetworkHelper checks connectivity before calling Firestore for saves, deletes, or initial syncs. Firestore deletes are best-effort, but the app always removes the workout from Room, so the UI never gets stuck waiting for the network to cooperate.
    • Timestamp-based conflict handling: When the user logs in and a userId is set, the app pulls workouts from Firestore and compares each item’s lastModified with the local copy. Only newer cloud data overwrites Room, which prevents stale data from another device from clobbering recent local edits.
  • What I focused on
    • Offline-first workout logging: Designing the app so adding and browsing workouts never depends on the network — Room is always the source of truth, and Firestore is treated as a sync/backup layer instead of the primary database.
    • Clear data modeling for goals & history: Structuring entities and DAO queries for workouts and workout goals so calendar views and progress screens can be driven from the same underlying data, instead of hard-coded calculations in the UI.
    • Separation of concerns & extensible code: Keeping Compose screens thin by pushing logic into ViewModels and repositories, wiring them with Flow/StateFlow, and aiming for small, clearly named functions so it’s easy to extend the app later with new workout metrics or additional health features without a rewrite.

Currency Converter

CurrencyConverter is a small Android app I built to practice clean MVVM, testing, and CI/CD around a simple, focused feature: converting money between currencies. Users can pick a source and target currency from dropdowns, type an amount, and get a formatted result using live exchange rates from a remote API. The app also crash free and handled all the exceptions or failed network calls.

  • Tech stack Kotlin, Jetpack Compose, MVVM, Hilt, Coroutines/Flow, Ktor + kotlinx.serialization, JUnit + MockK, GitHub Actions
  • Architecture
    • MVVM with Hilt-injected repository: A single CurrencyViewModel holds the screen state as a StateFlow<CurrencyUiState>. It doesn’t talk to the network layer directly; instead it depends on a CurrencyRepository interface provided by Hilt so the UI stays decoupled from API details.
    • Repository + Ktor API layer: The repository implementation (CurrencyRepositoryImpl) delegates HTTP calls to a CurrencyApiService built on Ktor’s HttpClient with ContentNegotiation and kotlinx.serialization.
    • Test-friendly design with fakes: Because the ViewModel depends on the CurrencyRepository interface, tests can swap in a FakeCurrencyRepository to simulate API results.
  • Reliability
    • Graceful handling of API failures: Network calls are wrapped in try/catch blocks inside the CurrencyViewModel, and any exceptions update the error field in CurrencyUiState instead of crashing the app.
    • Input validation & safe parsing: The ViewModel validates the input amount before converting and falls back to safe values (like "0.00") or an error message when the user types invalid text, so unexpected input doesn’t lead to inconsistent state or broken UI.
    • Single source of truth for UI state: All screen data (selected currencies, amount, result, and error) lives in a single CurrencyUiState exposed as a StateFlow.
  • What I focused on
    • Practicing production patterns on a tiny feature: Treating a simple currency conversion screen as a sandbox to practice MVVM, Hilt DI, a repository abstraction, and a clear separation between UI, state, and network code.
    • Designing for testability from the start: Depending on a CurrencyRepository interface in the ViewModel and using a FakeCurrencyRepository in tests so core behavior (conversion logic, error handling, and initial currency list) can be verified without real network calls.
    • End-to-end automation with CI/CD: Wiring up GitHub Actions workflows so each push runs the unit tests, and a separate pipeline builds a release APK and publishes it as a GitHub Release.
  • Testing & CI/CD
    • Unit tests for ViewModel logic: Wrote unit tests for CurrencyViewModel using a FakeCurrencyRepository and Kotlin coroutine test utilities. The tests verify happy-path conversions, invalid input handling, error state updates, and that the initial currency list is loaded as expected.
    • Network-free, deterministic tests: By depending on the CurrencyRepository interface instead of the real Ktor client, tests run quickly and reliably without hitting the actual API, which makes it easy to cover edge cases like invalid target currencies or exceptions.
    • Automated CI on each change: An Android CI GitHub Actions workflow runs ./gradlew testDebugUnitTest on every push and pull request to master, develop, and feature/** branches, so broken tests are caught before changes are merged.
    • Release pipeline via GitHub Actions: A separate Android CD workflow builds a release APK with ./gradlew assembleRelease and uploads it as a GitHub Release (tagged latest), giving me hands-on experience with a lightweight CI/CD pipeline for a small Android app.
